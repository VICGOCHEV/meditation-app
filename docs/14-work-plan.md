## Поток 1 — Бэкенд (этап 3 ТЗ, ~10 дней)

🟢 **Можем начать сразу.**

1. Поднять Node.js + PostgreSQL на тестовом VPS (отдельный systemd-сервис рядом с Caddy).
2. Спроектировать схему БД: `users`, `practices`, `user_progress`, `subscriptions`, `tracker_days`, `kt_history`, `bonus_unlocks`.
3. Написать API под существующие фронтовые контракты (`src/api/*.js`). Контракты уже есть — фронт переключится флагом `USE_MOCK=false` + `VITE_API_URL`.
4. Перенести логику прогрессии и бонусов с клиента на сервер (Zustand-сторы → API-вызовы; локально остаются только UI-флаги).
5. Аутентификация: email + пароль + bcrypt + JWT-сессии. Восстановление пароля по email.

🔴 **Зависимости от клиента:**
- Окончательные правила бонуса и «3 дня» (пункты `(а)`, `(б)` брифа) — без них не можем закладывать логику в БД.
- SMTP-провайдер для recovery-писем — нужен либо доступ к их аккаунту (Yandex 360 / SendPulse / Mailgun), либо разрешение использовать наш.

🟡 **Phone-auth (если делаем):** скаффолд страницы есть, нужен SMS-провайдер от клиента.

## Поток 2 — Хранение аудио + CMS (Strapi, этап 4 ТЗ)

🟢 **Можем начать сейчас.**

Поднимаем **Strapi** (open-source headless CMS на Node) как
отдельный процесс на том же VPS. Strapi даёт «из коробки» админку
с авторизацией, медиа-библиотеку с загрузкой файлов, content-type
builder и REST/GraphQL API — не нужно писать админку вручную.

**Архитектура:**

```
React-фронт (порт 8081, Caddy) ─────┐
                                    ├──► Caddy reverse-proxy
Custom Node backend (порт 3001) ────┤      ├─► /api/*  → custom backend
   • auth, sessions, JWT            │      ├─► /cms/*  → Strapi
   • user_progress, subscription    │      └─► /audio/* → Strapi uploads
   • ЮKassa webhooks                │
                                    │
Strapi CMS (порт 1337) ─────────────┘
   • Content types: Practice, Voice, Music
   • Media-library: загрузка mp3 / aac
   • Admin UI: логин/пароль
   • Лайфсайкл-хуки: auto-detect длительность mp3
```

**Content types в Strapi:**

| Type | Поля |
|---|---|
| `Practice` | `title` (string, required), `block` (enum: relaxation / awareness / author), `audio` (media, required), `duration_sec` (integer, **заполняется автоматически** из mp3-метаданных), `description` (text, optional), `price` (integer, optional — только для авторских), `published` (boolean), `order` (integer — для сортировки), `month_slot` (integer — для месячной ротации «Осознанности») |
| `Voice` | `name` (string: «Мужской» / «Женский»), `code` (enum: male / female), `audio_full` (media), `audio_preview` (media, 5–10 сек), `active` (boolean) |
| `Music` | `title` (string), `audio_full` (media), `audio_preview` (media, 5–10 сек), `active` (boolean), `order` (integer) |

**Что админ сможет делать сам через Strapi UI:**

1. **Фоновая музыка** — раздел «Music». Кнопка «Create new entry»:
   вписывает название, перетягивает mp3-файл (полная дорожка + 5–10 сек
   превью), нажимает «Save & publish». Трек сразу появляется в списке
   выбора музыки в онбординге.
2. **Голос проводника** — раздел «Voice». Может либо обновить
   существующий голос (заменить аудиофайл, оставив `code: male`), либо
   создать новый. У нас по дефолту два голоса (М и Ж), но при желании
   можно расширить — нужно будет только добавить значения в enum
   `code` или сделать его свободным string.
3. **Практики** — раздел «Practice». Выбирает блок (Расслабление /
   Осознанность / Авторский), вписывает название, перетягивает mp3,
   при необходимости — описание и цену для авторской. Длительность
   подтягивается автоматически: на сервере стоит лайфсайкл-хук
   `afterUpload`, который через npm-пакет `music-metadata` читает
   заголовки mp3 и пишет в `duration_sec`. Админ ничего вручную не
   считает.

**Что мы делаем под капотом:**

1. `cd /opt && npx create-strapi-app@latest cms --quickstart=false --skip-cloud`,
   выбираем PostgreSQL (та же база, что у нашего custom backend, но в
   отдельной схеме `strapi`).
2. Конфигурируем `config/database.js`, `config/server.js`.
3. Создаём через Content-Type Builder три типа: Practice, Voice, Music
   (можно делать прямо через UI — Strapi сразу пишет в код).
4. Кастомизация: добавляем lifecycle на Practice — `afterCreate` и
   `afterUpdate` распознают аудио и пишут duration.
5. Конфигурируем permissions: «Public» роль может только GET-ить
   published practices/voices/music; модификация — только Editor/Admin.
6. На фронте создаём `src/api/cms.js` — обёртка над Strapi REST:
   ```js
   GET /cms/api/practices?populate=audio&filters[block][$eq]=relaxation
   GET /cms/api/voices?populate=*
   GET /cms/api/music?populate=*
   ```
7. Меняем `mockPractices` на реальные ответы от Strapi. Флаг
   `VITE_USE_MOCK=false` — и пошло.
8. Аудиофайлы Strapi отдаёт по `/cms/uploads/<filename>` — Caddy уже
   делает reverse-proxy.
9. Логин админа Strapi — отдельный URL `/cms/admin`, первый
   суперюзер создаётся при `npm run develop` интерактивно.

**Доступ для админа:**
- URL: `http://212.43.148.208:8081/cms/admin` (после релиза — на боевом
  домене).
- Логин/пароль создаём один раз и передаём клиенту.
- Можно завести роли «Super Admin» (мы) и «Editor» (клиент) — Editor
  не может ломать модели данных, только редактировать контент.

🔴 **Зависимости от клиента:** только сами аудиофайлы (для наполнения).
До них CMS уже стоит, можно тренироваться на placeholder-mp3.

## Поток 5 — Telegram-бот + VK Mini App + push через них

🟡 **Можем подготовить инфраструктуру.**

1. **TG-бот**: написать сервисный код на сервере — webhook-эндпоинт, который принимает события «открылась практика / истекает подписка» из бэка и шлёт сообщения. Без токена не активен.
2. Адаптация UI под TG WebApp: theme params (использовать `tg.colorScheme`, `tg.themeParams`), `MainButton`/`BackButton`, `initDataUnsafe` для авторизации без логина.
3. То же для VK: `@vkontakte/vk-bridge`, `VKWebAppGetUserInfo` для авто-авторизации.

🔴 **Без чего не задеплоим в каталог:**
- TG-бот: токен от @BotFather.
- VK: App ID и привязанный домен.
- Для обоих обязателен HTTPS-домен (значит, нужно сначала закрыть вопрос с доменом и сертификатом).

## Поток 7 — Domain + HTTPS + боевой деплой

🔴 **Полностью заблокировано клиентом.**

Нужен купленный домен → подключить к серверу (DNS A-запись на `212.43.148.208`) → в Caddy поменять `:8081` на `meditation.example` → Caddy сам автоматом получит Let’s Encrypt-сертификат.

Без HTTPS:
- Web Push не работает (но мы и не делаем).
- Telegram Mini App не примет приложение (требует HTTPS).
- VK Mini App не примет.
- ЮKassa не пустит в боевой режим.

# Что делаем прямо сейчас (без клиента)

Параллельно стартуем три потока, каждый из которых может развиваться независимо:

**Сегодня-завтра (зелёные потоки):**
1. **Поток 4** (Yandex.Metrika) — самый быстрый: накатываю заглушку счётчика и развожу события по коду. Полдня.
2. **Поток 1.1–1.4** (бэкенд-каркас) — поднимаю Node + Postgres, схему БД, базовые API. 2–3 дня до состояния «фронт может переключиться с mock на real».
3. **Поток 6.1** (превью голоса/музыки) — раскрутить кнопки play в онбординге. Пара часов.

**Параллельно** заказчик присылает:
- одно аудио + одно превью голоса
- цену подписки
- ответы по бизнес-логике (а, б, в, г, д, е, з)
- тестовые ключи ЮKassa
- номер счётчика Метрики

Как только начнут приходить — разблокируется CMS-наполнение, ЮKassa, кастомизация условий бонуса.

Дальше (когда придёт домен и токены TG/VK) — пакуем в обёртки и отправляем на модерацию.
