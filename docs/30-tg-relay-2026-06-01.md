# TG-relay для обхода блокировки api.telegram.org

**Дата:** 2026-06-01
**Статус:** блокер для TG-пушей и /start-ответов бота
**Откуда вылез:** деплой webhook'а 01.06 на all-relaxme.ru → `setWebhook` упал с `fetch failed`

## Что сломано

С прод-сервера (Selectel `87.228.61.44`, Москва) **нет исходящего TCP/443 на `api.telegram.org`**:

```
ping 149.154.166.110          → 36ms OK (ICMP проходит)
TCP 443 → 149.154.166.110     → timeout
TCP 443 → google.com          → 200, 130ms (общий outbound норм)
TCP 443 → 149.154.167.220     → SSL handshake идёт (выборочно по IP)
```

Это **DPI/TSPU блокировка по конкретным IP телеграма**, не политика Selectel.
На любом RU-хостинге будет та же картина. Хостинг ни при чём.

## Что НЕ сломано

- **TG Mini App login** работает: фронт получает `Telegram.WebApp.initData`,
  бэк валидирует HMAC локально, исходящих в TG нет.
- **Persistent menu-button** уже стоит в BotFather — юзеры заходят в бота,
  жмут кнопку «Открыть приложение», попадают в Mini App. Это работает.

## Что сломано в применении к нам

- **`setWebhook`** не отработать (разовое, можно с любой машины вне РФ).
- **`sendMessage`** — любые пуши в личку юзера. Без этого нет:
  - welcome-ответа на `/start`
  - напоминаний о практике/замере DA
  - уведомлений о подписке (оплата прошла, истекает через 3 дня и т.д.)
  - любых триггер-сообщений

## Решение: Cloudflare Worker как relay

**Бесплатно, 5 минут, без аренды VPS, без админки.**

### Архитектура

```
backend (Selectel)
  └─ fetch → https://tg-relay.<account>.workers.dev/bot<TOKEN>/sendMessage
                  │ (edge вне РФ)
                  └─ https://api.telegram.org/bot<TOKEN>/sendMessage
```

### Код Worker'а (вся реализация)

```js
// Worker: tg-relay.workers.dev
// Прозрачный pass-through на api.telegram.org. Без логирования
// токена/payload'ов — безопасно.
export default {
  async fetch(req) {
    const u = new URL(req.url)
    const tg = await fetch(
      `https://api.telegram.org${u.pathname}${u.search}`,
      {
        method: req.method,
        headers: {
          'content-type':
            req.headers.get('content-type') || 'application/json',
        },
        body: ['GET', 'HEAD'].includes(req.method)
          ? undefined
          : await req.arrayBuffer(),
      }
    )
    return new Response(tg.body, {
      status: tg.status,
      headers: tg.headers,
    })
  },
}
```

### Изменения в нашем коде

[backend/src/utils/tgBot.js](../backend/src/utils/tgBot.js#L9):
```js
const API_BASE = process.env.TG_API_BASE || 'https://api.telegram.org'
```

В `/opt/meditation-app/backend/.env` на проде:
```env
TG_API_BASE=https://tg-relay.<account>.workers.dev
```

`systemctl restart meditation-api` → бот живой.

### Дополнительная защита

Worker открыт публично — любой, кто угадает URL + знает наш `TG_BOT_TOKEN`,
может слать через него запросы (но и без Worker'а мог бы — токен сам по себе
secret). Можно усилить:
- Добавить shared secret header (`X-Relay-Auth`), Worker валидирует.
- Поставить Cloudflare Access (бесплатно для small org).
Сейчас это overengineering — достаточно obscurity URL.

## Альтернативы (на случай если CF не подойдёт)

| Способ | Цена | Setup | Минусы |
|---|---|---|---|
| **CF Worker** ⭐ | 0 | 5 мин | зависим от CF (downtime раз в год) |
| Deno Deploy | 0 | 10 мин | менее устоявшийся |
| Vercel Edge Function | 0 | 10 мин | требует Git-репо |
| Hetzner CX22 + tinyproxy | 4€/мес | 30 мин | админка/обновления/firewall |
| Selectel VPS в Амстердаме | ~5€/мес | 30 мин | то же |
| TG Bot API local server | хостинг | 1 час | сам тоже стучится в TG — нужен relay под ним |

## План внедрения (когда решим)

1. Зарегать аккаунт Cloudflare (если ещё нет) — бесплатно, по email
2. Создать Worker через web UI: Workers & Pages → Create → Hello World template
3. Заменить код на 12 строк выше → Deploy
4. URL: `https://tg-relay.<account>.workers.dev` (видно сразу после deploy)
5. Локально с любой VPN-страны: проверить `curl <relay-url>/bot<TOKEN>/getMe` → должен вернуть `{"ok":true,"result":{...}}`
6. Запустить с локалки регистрацию webhook: `TG_BOT_TOKEN=... TG_WEBHOOK_URL=https://all-relaxme.ru/api/tg/webhook node backend/scripts/tg-set-webhook.js` (через relay чтобы не нужен был VPN — modify скрипт)
7. На проде: `echo "TG_API_BASE=https://tg-relay.<account>.workers.dev" >> /opt/meditation-app/backend/.env && systemctl restart meditation-api`
8. Тест: написать боту `/start` в Telegram → должен ответить с web_app кнопкой

## Какие пуши нужны (на будущее, после relay)

Ещё не обсуждали с клиентом, прикидываю по логике курса:
- [ ] Напоминание прослушать практику (1 раз в день в утра, если за вчера нет тракер-метки и есть активная подписка)
- [ ] «Завтра откроется следующая практика» (за день до окончания 4-дневного цикла)
- [ ] «Доступен промежуточный замер DA» (после 3-й практики)
- [ ] «Подписка истекает через 3 дня / истекла» (за 3 дня + день истечения)
- [ ] «Спасибо за оплату» (сразу после успешного YooKassa webhook'а)
- [ ] Welcome на `/start` — реализовано в коде, не работает из-за relay

Реализация — отдельный воркер (`backend/src/jobs/notifier.js`) с cron'ом
`node-cron`, дергает каждые 15 минут, шлёт через `tgBot.sendMessage`.
После relay не отличается от обычного HTTP-кода.
